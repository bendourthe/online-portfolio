What it is

One catalog, installed into every assistant you use

Nexus-Hub is the upstream catalog of skills, slash-commands, hooks, agents, and coding rules. An installer copies it into ~/.nexus-hub/ and wires it into your assistant's own config.

The same workflows then work identically across Claude Code, OpenAI Codex, Cursor, Gemini / Antigravity, GitHub Copilot, and the Nexus-AI desktop studio. Whatever tool a teammate prefers, they run the same governed commands and get the same disciplined output.

The five building blocks

CommandsSlash entry points like /describe that orchestrate a whole workflow.
SkillsSenior-level procedures the agent loads on demand (three-tier disclosure).
HooksDeterministic guardrails that fire on tool use - block secrets, guard destructive git.
AgentsSpecialized reviewer personas (security, performance, testing, ...).
RulesPer-language style & security standards applied automatically.

The difference

Why it makes any model dramatically more capable

Every assistant ships a handful of built-in commands and will happily answer a raw prompt. Nexus-Hub doesn't replace your model - it wraps it in the structure, depth, and guardrails a senior engineer brings. Same model, far better output.

Raw prompting & built-in commands

  • You re-explain the codebase from scratch every session
  • One-shot answers - no goal, no definition of done, no gate
  • Depth is whatever happens to be in the model's head that day
  • You trust the model to remember to be careful
  • Every engineer - and every tool - does it differently
  • Output you have to re-verify by hand

Nexus-Hub

  • Durable artifacts (analysis, review, plan) chain across sessions
  • Goal → definition-of-done → phased plan → GO / NO-GO gates
  • 252 expert skills load on demand, exactly when the task needs them
  • Hooks enforce safety the model cannot talk its way past
  • One governed workflow for the whole team, on every platform
  • Reviewable, traceable, accountable to a written constitution
★

It doesn't change the model - it changes what the model knows and does at each step. The same Opus, Sonnet, GPT, or Gemini that gives a generic answer to a raw prompt produces senior-level, reviewed, tested work when Nexus-Hub feeds it the right expert procedure at the right moment, holds it to a definition of done, and stops it from skipping steps. The gain compounds across the workflow - and because it's model-agnostic, it lifts whatever model you (or your teammate) happen to prefer.

Foundations

How Nexus-Hub actually works

Three things make this more than a prompt library: skills that load themselves, hooks that stop mistakes the model can't talk its way past, and governance that holds AI output to your written rules. Understand these and every command makes sense.

★

Skills that load themselves

252 skills · three-tier progressive disclosure

A skill is a senior-level procedure - "how to run a security review", "how to write idiomatic Go". You rarely invoke one by name; the agent auto-triggers the right skill from your task. The trick that makes 252 skills affordable is three-tier loading: only a tiny summary of every skill is always in context; the full procedure loads only when a skill is actually relevant.

TIER 1 · ALWAYS LOADED

Name + summary

~150-300 tokens per skill. Just enough for the agent to decide whether the skill is relevant. This is the only tier under token pressure.

TIER 2 · ON TRIGGER

The SKILL.md body

The working manual - step-by-step instructions, verification checklist, and rebuttals to shortcuts. Loaded only once the skill triggers.

TIER 3 · ON DEMAND

References + scripts

Long runbooks the agent reads only when needed, and scripts it executes without loading the source - heavy capability, zero context cost.

> add a FastAPI endpoint to export tasks as CSV
• matched skill fastapi-expert (tier-1 summary) → loading body
• matched skill api-design for pagination + error shape
(you didn't name either skill - the task did)
✓ applied async route, Pydantic response model, streaming CSV, 422 on bad filter

Manage the catalog with /skills: search <topic>, list, create, and import (which scans a skill for safety before installing - see Harden → Supply chain).

🔒

Hooks that stop mistakes

deterministic guardrails, enforced by the harness

Where skills are advice the model chooses to follow, hooks are enforced. A hook is a script that fires on a tool event (before or after the agent reads, writes, or runs something). The agent cannot rationalize past it - the harness simply blocks the action. This is the safety net that doesn't depend on the model remembering to be careful.

claude — taskflow
> (agent tries to hard-code the payment key back into config.py)
• Write config.py ...
⛔ secret-scan (PreToolUse hook) BLOCKED the write
line 12: hard-coded credential detected ( PAYMENT_KEY = "...<redacted>..." )
fix: load from an environment variable; never commit secrets
• agent reworks to os.environ["PAYMENT_KEY"] and retries
✓ Write config.py - clean, no secrets
> git reset --hard origin/main
⛔ git-guardrails (PreToolUse hook): destructive command
this discards all local changes. Confirm to proceed? [y/N]

Nexus-Hub ships 22 hooks - secret scanning, destructive-git confirmation, large-file guards, and consistency checks among them - registered in a single settings.json.

§

Governance that keeps AI aligned

/constitution · /spec

The biggest risk with AI coding isn't a typo - it's the agent confidently doing something that violates a team norm nobody wrote down. Nexus-Hub fixes that by letting you write the norms down and check work against them.

/constitution

Author a versioned governance document of MUST / SHOULD principles your team agrees on. Every /plan runs a Constitution Check against it - so plans that would violate a principle are caught before code is written (you saw this gate pass on the Plan page).

/spec

Clarify an underspecified request before coding (drive out the ambiguities), or analyze a feature directory for cross-artifact consistency and coverage. Stops the "we built the wrong thing because the spec was vague" failure.

☷ .specify/memory/constitution.md authored by /constitution

TaskFlow Constitution - v1.0

MUST

  • All SQL uses parameterized queries - never string interpolation
  • No secret is ever committed to source
  • Every merge to develop passes CI with coverage ≥ 80%

SHOULD

  • Prefer reverse-engineering a capability over adding a dependency
  • Keep functions under 40 lines; extract a service layer where routes touch the DB directly

Plans, specs, and reviews all align to this document. Amendments are versioned.

★

This is how "the AI did something weird" becomes reviewable. When work is held to written MUST/SHOULD principles, a deviation is a concrete, catchable violation - not a judgment call. The constitution makes AI output accountable to the same rules as the humans on the team.

Workflow 02 · Planning the work

Turn findings into a phased plan

You know what TaskFlow is and where it hurts. Now decide what to do - and capture it as a robust, gated plan instead of a vague intent. There are two on-ramps to /plan: start from your own findings, or mine an external project for ideas first with /compare.

You have the review report. Plan the remediation.

Point /plan at docs/0.4.0/review-full.md and it ingests the findings as the raw material for a phased plan - security-first ordering, a definition-of-done, and gates baked in.

/plan always runs a goals-first step before decomposing: it fixes the target problem, the persona, and an observable definition-of-done. A detailed plan for the wrong objective is the most common planning failure, and this step prevents it. It then runs a Constitution Check and Complexity Tracking gate, and emits tasks in a strict T### format.

Phase 2 - CI/CD pipeline

★

Phases are the unit of work. The plan is deliberately decomposed so each phase is small enough to implement, test, and commit on its own branch. That's what keeps diffs reviewable and the git history clean when you reach /implement.

You found an open-source repo worth learning from.

An OSS task manager, TaskMaster, has excellent recurring-task and reminder logic. Rather than guess what's worth adopting, /compare does a deep, structured comparison and writes a gap analysis with a prioritized adoption plan.

Reverse-Engineering assessment (excerpt)

The recurring-task logic is a pure scheduling algorithm over the dateutil.rrule standard - it carries no external service and no proprietary code. Recommendation: reimplement the algorithm in backend/app/recurring.py rather than depending on TaskMaster. This keeps TaskFlow's dependency surface flat and avoids licensing/attribution coupling.

Security assessment (excerpt)

No adoption candidate introduces an outbound call, a credential, or a new third-party runtime dependency. All three are local algorithms. Verdict: safe to reverse-engineer.

★

Reverse-engineer-first is a policy, not a vibe. Nexus-Hub's MCP & adoption policy walks a decision tree - local-only → LLM-native skill → reverse-engineer into local code → trusted-vendor wrapper → drop. /compare applies that tree to every candidate, so "what can we learn from this repo" never silently becomes "add five dependencies and ship our source to a vendor".

→

It chains straight into a plan. When the comparison finishes, /compare offers /plan from-comparison, which seeds a phased plan directly from the adoption table - with reverse-engineer-first ordering. Same destination as Path A: a gated plan ready for /implement.

TaskFlow - Project Analysis

Generated by /describe full · version 0.4.0 · mode: software project

1. Overview

TaskFlow is a multi-tenant SaaS for team task management: workspaces, projects, tasks, assignments, comments, and a (partial) billing layer. Users authenticate with email + password and receive a JWT. The product is pre-1.0 and several flagged areas are incomplete.

2. Tech stack

  • Backend: Python 3.11, FastAPI, SQLAlchemy, Alembic, Pydantic v2, Redis (RQ queue)
  • Frontend: React 18, TypeScript 5, Vite, TanStack Query, Tailwind
  • Data: PostgreSQL 15; Infra: docker-compose (no Kubernetes, no CI/CD)

3. Structure & key modules

  • backend/app/api/ - 6 route groups: auth, tasks, projects, search, billing, webhooks
  • backend/app/models/ - SQLAlchemy ORM models (referenced directly by routes - tight coupling)
  • backend/app/config.py - settings & secrets loading
  • frontend/src/ - pages/, components/, api/ (typed client)

4. Data flow

SPA → /api/* (JWT bearer) → FastAPI route → ORM model → Postgres. Background jobs (email, billing reconcile) are pushed to Redis and processed by an RQ worker.

5. How to run

  • docker compose up brings up api + worker + postgres + redis
  • Frontend dev server: cd frontend && npm run dev

  • Billing endpoints return 501 Not Implemented - feature is a stub
  • No .github/workflows - nothing runs on push
  • config.py appears to contain a hard-coded credential - verify in /review security
  • Test directory is sparse - coverage looks low

Workflow 03 · Implementation

Build the plan, one phase at a time

A plan is only as good as the discipline used to execute it. /implement takes a single phase from end to end - review, code, lint, test, troubleshoot, gate, document, commit - and refuses to advance until the quality gate passes. The result is a clean git history where every commit is a complete, tested unit of work.

Implement Phase 1 of the harden-and-complete plan

You run /implement. It discovers the plan, asks which phase (defaulting to the first incomplete one), and offers to create a feature branch before writing any code.

What just happened (the discipline)

The per-phase workflow

Every phase runs the same sequence, so output quality doesn't depend on the engineer remembering the steps:

  1. Resolve plan + phase
  2. Offer a feature branch
  3. Pre-implementation review
  4. Implement subtask by subtask
  5. Lint + format
  6. Run tests + coverage
  7. Troubleshooting loop (fix until green)
  8. GO / NO-GO quality gate
  9. Post-phase: docs, devlog, commit, push

The branching model

Each phase lands on its own short-lived feat/* branch off develop, keeping the git graph legible and every phase independently reviewable:

main v0.4.0
develop integration
feat/security-fixes
fix(security): parameterize search ...
merge feat/security-fixes
feat/ci-pipeline (phase 2)
★

NO-GO actually stops it. The quality gate is not decorative. If lint fails, a test is red, or the change drifts outside the phase's scope, /implement enters a troubleshooting loop and will not run the commit step. You never get a half-broken phase quietly committed - which is the failure mode unstructured AI coding produces most.

⚑

The final phase ships itself. When the phase you implement is the last phase of the plan, /implement auto-detects it and runs release readiness, handing off to /update release (covered in Workflow 05). The plan literally drives the project from "inherited" to "tagged release".

Workflow 04 · Hardening

Make it secure and well-tested

Two pillars of production-readiness, two commands. /review finds the problems (read-only); /test closes the coverage gap and wires the CI gate that keeps it closed. This is where TaskFlow's CRITICAL SQLi and its 34% coverage get dealt with for good.

/review security runs a focused audit - secrets, auth, input validation, dangerous patterns - and reports. /review pentest goes deeper: an OWASP-WSTG-aligned penetration test that produces proof-of-concept findings, not just warnings. Both are read-only; the fixes happen under /implement (which is exactly what Phase 1 did).

TaskFlow - Penetration Test

/review pentest · OWASP WSTG · proof-of-concept findings

F-01 · SQL Injection (CRITICAL)

Endpoint: GET /api/search   CWE-89. The q parameter is interpolated into a raw SQL string. A crafted payload bypasses tenant isolation and dumps all rows.

  • PoC: ?q=a' OR '1'='1' --
  • Fix: use a parameterized query / ORM filter (Phase 1 / T001)
  • Verify: regression test asserting the payload returns 0 cross-tenant rows (T003)

F-02 · Hard-coded secret (HIGH)

Live payment key committed at config.py:12. Rotate the key at the provider, load from an env var, and purge it from git history (Phase 1 / T002).

F-03 · Weak JWT validation (MEDIUM)

Tokens are decoded without verifying aud / iss. Add claim validation to the auth dependency.

⚠

Find here, fix elsewhere. Even a pentest is read-only. That's intentional: you want an untainted record of what was exploitable, separate from the commits that fixed it. The report's CWE IDs and PoCs map one-to-one onto plan tasks, so the audit trail from "found" to "fixed and tested" is complete.

Your running code is only part of the attack surface. Third-party dependencies ship CVEs, and any skill you import into the catalog runs inside your agent's context. Three read-only scopes cover the supply side: a dependency CVE / license scan, an SBOM for compliance, and a pre-install security scan for skills.

Generated tests follow the repo's standards automatically - AAA structure, parametrized cases, fixtures over inline setup, no sleeps.

Workflow 05 · Ship & maintain

Sync everything, then ship it

All four findings resolved, coverage at 82%, CI green. The last mile is making sure documentation reflects reality and cutting a clean release. /update owns every "bring the repo up to date" action - and at release scope, runs the whole ship flow as one gated sequence.

/update docs

Sync the docs to reality

Refreshes README, architecture docs, and inline guides to the repo's current state - headline counts, a "what's new" narrative, and a link/staleness audit. Not just version-string bumps.

/update refactor

Tidy the layout

Reorganizes docs/ and project artifacts to convention and archives prior-version files - with full reference repair so nothing dangles.

/update release

Cut the release

The full flow: docs → devlog → gitignore → version → changelog → refactor, then commit, tag, push. Atomic, drift-guarded, gated.

Ship v0.5.0

/update release runs the focused scopes in order, then commits, tags, and pushes - never creating a tag or pushing without explicit confirmation. The version bump is atomic across every version-carrying surface and verified by a drift guard.
